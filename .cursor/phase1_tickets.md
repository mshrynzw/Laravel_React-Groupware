# Phase1 実装チケット（案B）

## 前提（確定事項）

- 対象範囲: 認証（Sanctum）+ ユーザー管理 + グループ管理
- 優先順位: ログイン/ログアウト -> ユーザーCRUD -> グループCRUD
- ロール: `superadmin` / `admin` / `member`
- 権限範囲: `admin` は自分の所属グループのみ操作可
- 認証方式: Sanctum SPA Cookie 認証
- 認証機能: パスワードリセット・メール認証を含む
- 所属モデル: ユーザーは複数グループ所属可
- 主グループ: `primary_group_id` は NULL 可
- テスト方針: 案1（Feature中心）で開始し、案2（Feature + Unit）へ段階移行
- 用語統一: 「組織」は使わず「グループ」に統一

---

## DB チケット

### DB-01: ロール定義
- **目的**: ロールベース認可の土台を作る
- **作業内容**:
  - `users.role` を追加（`superadmin/admin/member`）
  - デフォルト値と既存データ移行方針を定義
- **受け入れ条件**:
  - 全ユーザーが有効ロールを持つ
  - 不正ロールが保存されない

### DB-02: グループテーブル作成
- **目的**: グループ管理の永続化
- **作業内容**:
  - `groups` テーブル作成（name, description, created_by, updated_by, timestamps）
  - 一意制約方針を定義（name）
- **受け入れ条件**:
  - グループ作成/更新/削除が可能
  - 重複ルールが期待通りに効く

### DB-03: ユーザー・グループ中間テーブル
- **目的**: 複数グループ所属を表現
- **作業内容**:
  - `group_user` テーブル作成（`user_id`, `group_id`）
  - 複合ユニーク制約 (`user_id`, `group_id`)
  - 外部キー制約と削除時挙動を設定
- **受け入れ条件**:
  - 同一所属の重複登録が防止される
  - 参照整合性が維持される

### DB-04: 主グループ対応
- **目的**: 主グループを任意指定可能にする
- **作業内容**:
  - `users.primary_group_id`（nullable）追加
  - FK 削除時 `SET NULL` を設定
- **受け入れ条件**:
  - 主グループ未設定（NULL）で運用可能
  - 指定済み主グループ削除時に NULL 化される

### DB-05: 認証補助テーブル確認
- **目的**: メール認証/パスワードリセットの成立性担保
- **作業内容**:
  - `password_reset_tokens`・`email_verified_at` 運用確認
- **受け入れ条件**:
  - 認証メール・リセットメールの基盤が有効

### DB-06: 初期データ Seeder
- **目的**: 開発/検証環境の立ち上がりを高速化
- **作業内容**:
  - `superadmin` 初期ユーザー
  - 検証用グループ/管理者ユーザー投入
- **受け入れ条件**:
  - seed 実行後に基本操作をすぐ検証できる

## API チケット（Laravel）

### API-01: Sanctum SPA Cookie 認証初期設定
- **目的**: SPA 認証の基盤構築
- **作業内容**:
  - CORS/stateful domain/CSRF 設定
  - `/sanctum/csrf-cookie` -> login の接続確認
- **受け入れ条件**:
  - フロントからセッションログイン可能

### API-02: 認証エンドポイント
- **目的**: 基本認証フローを提供
- **作業内容**:
  - `POST /login`, `POST /logout`, `GET /me`
- **受け入れ条件**:
  - ログイン状態取得とログアウトが安定動作

### API-03: メール認証
- **目的**: メール認証フロー実装
- **作業内容**:
  - 認証メール送信・認証実行・再送
- **受け入れ条件**:
  - 未認証 -> 認証済みへ遷移できる

### API-04: パスワードリセット
- **目的**: メール経由の再設定フロー実装
- **作業内容**:
  - リセットメール送信・トークン検証・再設定
- **受け入れ条件**:
  - リセット完了後に新パスワードでログイン可能

### API-05: ユーザーCRUD
- **目的**: ユーザー管理の中核機能
- **作業内容**:
  - `index/show/store/update/destroy`
  - 一覧の名前/メール検索・ページネーション
  - 作成時に仮パスワードを設定
- **受け入れ条件**:
  - 主要ユースケースが API 経由で完結

### API-06: ユーザー所属グループ操作
- **目的**: 所属管理と主グループ更新
- **作業内容**:
  - 所属追加/削除
  - `primary_group_id` 更新（NULL 許容）
- **受け入れ条件**:
  - 複数所属・主グループ未設定が扱える

### API-07: バリデーション/重複防止
- **目的**: 入力品質と整合性担保
- **作業内容**:
  - メール一意、必須項目、ロール妥当性
  - 重複所属の防止
- **受け入れ条件**:
  - 不正入力で適切なエラー応答

### API-08: グループCRUD
- **目的**: グループ管理機能提供
- **作業内容**:
  - `index/show/store/update/destroy`
  - 一覧検索（name）
- **受け入れ条件**:
  - グループ管理の主要操作が可能

### API-09: グループ所属メンバー参照
- **目的**: グループの所属可視化
- **作業内容**:
  - グループ詳細で所属ユーザー取得
- **受け入れ条件**:
  - グループ単位で所属者を確認できる

### API-10: 認可（Policy/Gate）
- **目的**: ロール別アクセス制御
- **作業内容**:
  - `superadmin`: 全グループ対象
  - `admin`: 所属グループのみ
  - `member`: 自己関連中心
- **受け入れ条件**:
  - 非許可操作が 403 で拒否される

### API-11: 最低限の監査情報
- **目的**: 変更追跡可能性の確保
- **作業内容**:
  - `created_by`/`updated_by` もしくは操作ログ記録
- **受け入れ条件**:
  - 作成・更新操作の実行者が追跡できる

## React チケット

### FE-01: ログイン/ログアウトUI
- **目的**: 認証導線の提供
- **受け入れ条件**:
  - ログイン後に保護画面へ遷移、ログアウトで解除

### FE-02: メール認証UI
- **目的**: 認証待ち/再送/完了遷移の提供
- **受け入れ条件**:
  - 未認証ユーザーが迷わず認証完了できる

### FE-03: パスワードリセットUI
- **目的**: 再設定導線の提供
- **受け入れ条件**:
  - メール送信から再設定完了まで実行できる

### FE-04: ユーザー一覧UI
- **目的**: ユーザー検索/閲覧の提供
- **受け入れ条件**:
  - 名前/メール検索とページネーションが機能

### FE-05: ユーザー作成/編集/削除UI
- **目的**: 管理操作の提供
- **受け入れ条件**:
  - CRUD と仮パスワード運用が画面で完結

### FE-06: ユーザー所属グループ編集UI
- **目的**: 複数所属と主グループ操作
- **受け入れ条件**:
  - 所属追加/削除と主グループNULL設定が可能

### FE-07: グループ一覧/CRUD UI
- **目的**: グループ運用の提供
- **受け入れ条件**:
  - CRUD とエラー表示が適切

### FE-08: グループ詳細（所属メンバー）UI
- **目的**: グループ単位での可視化
- **受け入れ条件**:
  - 所属メンバー情報を確認できる

### FE-09: 権限ベース表示制御
- **目的**: 誤操作予防
- **受け入れ条件**:
  - ロールに応じて操作UIが適切に制御される

### FE-10: 共通エラーハンドリング
- **目的**: UX と運用性の向上
- **受け入れ条件**:
  - 401/403/419/validation エラーを統一表示
