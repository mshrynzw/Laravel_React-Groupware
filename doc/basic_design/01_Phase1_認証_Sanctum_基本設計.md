## 1. 目的・スコープ

SPA（React）と Laravel API 間で、セキュアなログイン状態を維持する。Sanctum を用いたトークン／セッション方式の認証、CSRF 対策、フロントのログイン画面および認証ガードを対象とする。

---

## 2. 機能概要

| 項目 | 内容 |
|------|------|
| Sanctum 導入 | API 用認証ドライバ設定、CORS・ミドルウェア連携 |
| ログイン API | 資格情報検証、成功時にトークンまたはセッション確立 |
| ログアウト API | トークン失効／セッション破棄 |
| セッション管理 | 有効期限・再発行方針（必要に応じてリフレッシュ） |
| CSRF 対応 | Cookie ベース時は CSRF トークン取得・検証フロー |
| フロントログイン画面 | 入力フォーム、エラー表示、ログイン後リダイレクト |
| 認証ガード（Route Guard） | 未認証時の保護ルートからのリダイレクト |

---

## 3. 画面・API（概要）

### 3.1 画面

- ログイン画面（ID／パスワード、エラーメッセージ表示）

### 3.2 API（例）

| メソッド | パス（案） | 説明 |
|----------|------------|------|
| POST | `/api/login` | ログイン |
| POST | `/api/logout` | ログアウト |
| GET | `/api/user` または `/api/me` | 認証ユーザー情報取得 |

※ 実際のパスは API 設計書と整合させる。

---

## 4. データモデル

- 認証主体は既存の `users` テーブル（ユーザー管理で定義）を利用。
- 本ステップ単体では新規テーブルは必須ではない（Sanctum の `personal_access_tokens` 等はマイグレーションで追加）。

---

## 5. 非機能・セキュリティ

- パスワードはハッシュ保存（Laravel 標準）。
- HTTPS 前提での Cookie 運用。
- レート制限（ログイン試行）を検討。
- **監査ログ**: ログイン成功・失敗、ログアウト、パスワード変更やトークン失効など、認証状態に関わる操作について、**誰が・いつ・どの操作を行ったか**を記録する（失敗ログはブルートフォース対策と個人情報保護のバランスを取る）。

---

## 6. 依存関係

- **前提**: ユーザーレコードが存在すること（ユーザー管理と連携）。
- **後続**: 全機能の認可チェックの基盤となる。
