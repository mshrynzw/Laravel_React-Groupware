## 1. 目的・スコープ

システム利用者の登録・参照・更新・削除と、ロールに基づく権限制御（RBAC）の基盤を提供する。

---

## 2. 機能概要

| 項目 | 内容 |
|------|------|
| users テーブル | ログイン ID、表示名、メール、パスワードハッシュ等 |
| ユーザー CRUD API | 一覧・詳細・作成・更新・削除（権限に応じ制限） |
| ロール管理（roles） | ロール定義テーブル、ユーザーとの紐付け |
| 権限制御（RBAC） | ポリシー／ミドルウェア／フロント表示制御 |
| ユーザー一覧画面 | 検索・ページング、管理者向け操作 |

---

## 3. 画面・API（概要）

### 3.1 画面

- ユーザー一覧（管理者）
- ユーザー詳細／編集（権限に応じ）

### 3.2 API（例）

| メソッド | パス（案） | 説明 |
|----------|------------|------|
| GET | `/api/users` | 一覧 |
| GET | `/api/users/{id}` | 詳細 |
| POST | `/api/users` | 作成 |
| PUT/PATCH | `/api/users/{id}` | 更新 |
| DELETE | `/api/users/{id}` | 削除 |

---

## 4. データモデル（案）

| テーブル | 主な項目 |
|----------|----------|
| `users` | id, name, email, password, … |
| `roles` | id, name, slug |
| `role_user`（または同等） | user_id, role_id |

※ 組織との関連は組織管理ステップの `user_company` と整合させる。

---

## 5. 非機能・セキュリティ

- 管理者のみユーザー作成・削除可能とする等、RBAC を API・画面の両方で適用。
- 一覧の個人情報取り扱いに注意（表示項目の最小化）。
- **監査ログ**: ユーザーの作成・更新・削除、ロール付与・剥奪について、**誰が・いつ・どのユーザーに対して・何を変更したか**を記録する。

---

## 6. 依存関係

- **前提**: 認証（Sanctum）が利用可能であること。
- **連携**: 組織管理、勤怠・ワークフロー等の「誰が操作できるか」の判定。
