## 1. 目的・スコープ

会社・部署などの階層構造を表現し、ユーザーとグループの所属関係を管理する。ツリー表示による閲覧を対象とする。

---

## 2. 機能概要

| 項目 | 内容 |
|------|------|
| companies テーブル | グループ単位（会社・事業部等）の定義 |
| groups テーブル | 部署等、親子関係でツリー表現可能な設計 |
| user_company 関連 | ユーザーとグループ（・部署）の多対多または所属履歴 |
| グループツリー表示 | API でツリーまたはフラット＋パス、フロントで折りたたみ表示 |

---

## 3. 画面・API（概要）

### 3.1 画面

- グループツリー閲覧（管理者向け編集を含めるかは要件次第）

### 3.2 API（例）

| メソッド | パス（案） | 説明 |
|----------|------------|------|
| GET | `/api/companies` | グループ一覧／ツリー |
| GET | `/api/groups` | 部署一覧／ツリー |
| （必要に応じ） | CRUD | 管理用エンドポイント |

---

## 4. データモデル（案）

| テーブル | 主な項目 |
|----------|----------|
| `companies` | id, name, parent_id（任意）, … |
| `groups` | id, company_id, name, parent_id, … |
| `user_company` | user_id, company_id, department_id（任意）, … |

---

## 5. 非機能

- 階層の深さ上限、循環参照防止の制約を DB またはアプリで検討。
- **監査ログ**: 会社・グループ・ユーザー所属の作成・更新・削除について、**誰が・いつ・どのリソースに対して・何を変更したか**を記録する。

---

## 6. 依存関係

- **前提**: `users` が存在すること。
- **連携**: ワークフロー承認者の指定、勤怠の部門別集計等で利用可能。
