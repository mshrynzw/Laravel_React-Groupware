---

## 1. 目的・スコープ

機能領域をまたいで共通となる API 設計方針、エラー処理、ログ、権限、バリデーション、テストの基準を定め、各ステップの基本設計と矛盾なく実装できるようにする。

---

## 2. 機能・方針概要

| 項目 | 内容 |
|------|------|
| API 設計 | REST のリソース命名、バージョニング、ページング形式（cursor / offset）、共通レスポンスラッパ |
| エラーハンドリング | HTTP ステータス、業務エラーコード、フロント表示用メッセージのマッピング |
| ログ管理 | リクエスト ID、レベル、個人情報マスキング、**監査ログ（下記）** |
| 権限制御 | RBAC とポリシーの適用範囲（API ミドルウェア・コントローラ・フロント） |
| バリデーション | Form Request／OpenAPI とフロントの同期方針 |
| テスト | 優先度（後回し可）、単体／API／E2E の対象範囲 |

---

## 3. 共通 API 規約（案）

- 成功時: `200` / `201`、一覧は `meta` にページ情報を含めるか統一。
- 失敗時: `422`（バリデーション）、`403`（権限）、`401`（未認証）、`404`。
- エラーボディ例: `{ "message": "...", "errors": { ... } }`。

---

## 4. 非機能

- レート制限、CORS、本番のログ集約先（ファイル／クラウド）。

### 4.1 監査ログ（操作履歴）— 全機能共通

各領域のデータについて、**作成・更新・削除**および業務上重要な**状態変更**（公開切替、承認、計算実行など）が発生したとき、**誰が（操作ユーザー）・いつ・どのリソースに対して・何を変更したか**を追跡できるよう監査ログ（または等価な履歴テーブル・イベントログ）に残す。領域ごとの対象範囲は各 `doc/basic_design/` の非機能に記載する。
